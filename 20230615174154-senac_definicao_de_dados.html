Definição de dados

Na computação, dados são toda a informação que pode ser traduzida eficientemente para a leitura, a transferência e o processamento.

  • Todos os dados precisam ser armazenados de alguma maneira que se possa acessa-los e altera-los
  • A unidade atomica de informação é o bit, capaz de representar um sim ou um não, um 0 ou 1, verdadeiro ou falso.

Medidas comuns de armazenamento de dados

Unidade Valor
bit 1 bit

kilobyte 1024 bytes

  • Cada tipo de dados (Inteiro, booleano, String, real, decimal) ocupa uma quantidade diferente de dados na memória.

Criação de um banco de dados

Um banco de dados nada mais é do que uma coleção de dados organizados de modo que seja possível efetuar buscas e transações fácil e rapidamente, mesmo quando se está trabalhando com grandes quantidades de dados. No contexto técnico, isso significa que a principal função do banco de dados é interagir com os sistemas a ele conectados, capturar, analisar, calcular e responder rápida e organizadamente.

Criando o banco de dados com SQL

Conectando-se ao banco de dados

  • Inicalmente é necessário conectar-se com o banco de dados. Podemos fazer isso de duas formas:
    • Command line interface (CLI)

      • Para conectar-se ao banco de dados MySQL pelo terminal podemos usar o comando

        mysql -u root -p

        Ou (utilizando no Arch Linux)

        mariadb -u root -p

      • A partir dai o terminal estará conectado ao banco e o novo prompt do shell será

[MARIADB (NONE)>]
  • A partir daqui podemos executar comandos SQL
  • MySQL workbench
    • Conectar-se pela interface

Utilizando SQL

  • Inicialmente não há databases (além das padrões do software) criadas. O comando SQL para criar uma database é:

    CREATE DATABASE nome_do_banco

  • Exemplo:

    CREATE DATABASE farmacia

  • Após esse comando a database farmácia foi criada, podemos verificar com o comando:

    SHOW DATABASES;

  • Ou através da aba "schemas" no mysql workbench;

  1. Usando o banco de dados criado

    • Para usar o banco de dados criado utilize o comando:

      USE nome_do_banco

Criando tabelas

  • A função CREATE TABLE recebe o nome de uma coluna, seguido de seu tipo e marcadores opcionais (primary key, unique, not null, auto increment), elementos separados pro virgula, tendo como uso genério a seguinte forma:

      CREATE TABLE nome_tabela (
nome_campo1 TIPO REGRAS,
nome_campo2 TIPO REGRAS,
nome_campo3 TIPO REGRAS,     
     )

  • Exemplo:

    USE farmacia;
CREATE TABLE produtos (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT 
   nome VARCHAR(20) NOT NULL,
   marca VARCHAR(20),
   preco DECIMAL(9,2),
   generico BOOLEAN,
   validade DATE
   )

  1. Principais regras para colunas:

    1. NULL ou NOT NULL

      • Define se uma coluna pode armazenar valores nulos. Permitir valores nulos significa que o campo não é obrigatório:
        CREATE TABLE generic (
coluna_1 VARCHAR(20) NOT NULL, -- campo obrigatório
coluna_2 INT NULL, -- campo não obrigatório
coluna_3 DOUBLE -- não especificar "NOT NULL" em um campo o torna imediatamente não obrigatório, isso é, pode armazenar valores nulos
  );

      • Extra-texto — Exemplo: Suponha que um jogador criou uma conta, mas ainda não criou seu personagem

        CREATE TABLE (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   username VARCHAR(20) NOT NULL UNIQUE,
   personagem VARCHAR(20)
   )
        • Nesse caso, se o campo personagem fosse "NOT NULL" um erro ao criar a conta seria gerado, pois o personagem ainda não foi criado, dessa forma devemos manter o personagem como NULL pois o jogador pode criar a conta antes de criar o personagem.

    2. DEFAULT

      • A regra DEFAULT define um valor padrão para uma coluna

        coluna1 CHAR(1) NOT NULL DEFAULT('S')

      • Considere o caso anterior, mas agora o jogador pode comprar a versão premium do jogo, no entanto a conta começa como uma conta gratuita:

        CREATE TABLE (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   username VARCHAR(20) NOT NULL UNIQUE,
   personagem VARCHAR(20)
   premium BOOLEAN NOT NULL DEFAULT(FALSE)
   )

    3. AUTOINCREMENT

      • Torna a coluna sequencial, isso é, uma coluna gerida pelo próprio banco de dados que, a cada novo registro, incrementará em 1 o valor do campo, iniciando em 1 e seguindo o valor da entrada anterior:

        • Suponha que duas contas foram criadas: "usuer1" e "user2" são seus usersnames. A tabela ficaria assim:

        | id | user | personagem | premium |

        1 user1 NULL FALSE
        2 user2 NULL FALSE
        • Sendo que o campo ID foi incrementado automaticamente, o campo personagem pode ser nulo e o valor padrão de "premium" inicializa como falso

    4. PRIMARY KEY

      • Define um campo como chave primária

Excluindo tabelas

  • Para excluir tabelas utilizamos o comando:

    DROP TABLE nome_da_tabela

Inserindo dados em uma tabela

  • Para inserir dados em uma tabela utilizamos o comando INSERT INTO

    INSERT INTO nome_tabela (campo1, campo2, campo3) VALUES ("valor1", "valor2", "valor3")

Consulta simples aos dados de uma tabela

SELECT * FROM nome_da_tabela;

  • O comando SELECT indica que vamos ler dados, o token * significa que buscamos todos os campos disponíveis (pode ser traduzido como "TODOS"), o comando FROM especifica a tabela que estamos consultando, sendo a forma genérica do comando:

    SELECT campo(s) FROM tabela

Alterando colunas da tabela

  • para alterar o formato de uma tabela podemos utilizar o comando ALTER

    ALTER TABLE nome_tabela AÇÃO

  1. Adicionando novos campos

    ALTER TABLE nome_tabela ADD novo_campo TIPO REGRAS

    • Exemplo:

      ALTER TABLE remedios ADD valilade DATE NOT NULL

  2. Modificando um campo existente

    ALTER TABLE nome_tabela MODIFY COLUMN coluna TIPO RESTRIÇÕES

    • Exemplo:

      ALTER TABLE remedios MODIFY COLUMN validade DATETIME NOT NULL

  3. Excluido um campo

    ALTER TABLE nome_tabela DROP COLUMN coluna;

    • Exemplo:

      ALTER TABLE remedio DROP COLUMN validade;

Tipos de dados

Como visto anteriormente, ao definir uma coluna, é necessário estabelecer o tipo de informação que ela deve carregar. A SQL estabelece alguns tipos próprios para números, texto, entre outros. Além disso, os SGBDs podem trazer variações dos tipos padrão ou tipos novos.

  • OBS: Serão vistos alguns tipos padrão e outros implementados pelo MYSQL

Tipos numéricos

  • TINYINT: 1 byte
    • numeros inteiros de -128 a 127
    • É possível usar o modificador UNSIGNED (sem sinal) para especificar que a coluna receberá apenas números positivos. Nesse caso, a faixa de valores se torna de 0 a 255
  • SMALLINT 2 bytes
    • numeros inteiros de 32768 a +32767
    • Número inteiro com faixa pequena de valores. Com o modificador UNSIGNED, a faixa de valores fica de 0 a 65535.
  • INT: 4 bytes
    • Numeros de -2147483648 a +2147483647
    • Número inteiro. É o tipo mais usual, apesar de sua faixa de valores bastante ampla. A versão UNSIGNED amplia seu valor positivo máximo a 4294967295.
  • BIGINT: 8 bytes
    • numeros de -9223372036854775808 a 9223372036854775807
    • Utilizado em situações bastante específicas em que a coluna precisa armazenar números muito altos ou muito baixos. Deve ser usado com cuidado para não desperdiçar memória. Também conta com versão UNSIGNED, chegando a suportar o valor 18446744073709551615.
  • DECIMAL(m,n) ou NUMERIC(m,n): tamanho variavel e sem limite
    • Numero com ponto fixo e valor exato. Define precisão por M (quantidade máxima de digitos do valor armazenado) e quantidade exata de digitos decimais por n. Sendo assim armazena no máximo m digitos sendo n digitos decimais
  • FLOAT: 4 bytes
    • Ponto flutuante e valor aproximado
  • DOUBLE: 8 bytes
    • Ponto flutuante com maior precisão e valor aproximado
  • BIT(n): até 8 bytes
    • Representa valores binários, o parametro n representa a quantidade de bitos que o valor armazenará, podendo ir de 1 a 64. (Default = 1)

Tipo lógico

  • BOOLEAN: armazena valores 0 ou 1 - é por baixo dos panos um TINYINT

Tipos textuais

  • CHAR(n) texto com tamanho fixo definido por n. Se o valor cadastrado for menor do que n preenche o restante com espaços em branco
  • VARCHAR(n): texto com tamanho variavel e tamanho máximo n. Se o valor cadastradado for menor que n comprime o espaço
  • TEXT: texto de tamanho variavel e praticamente ilimitado
    • Na realidade, uma coluna do tipo TEXT pode guardar até 65535 bytes de dados. Há variações desse tipo: TINYTEXT (255 caracteres), MEDIUMTEXT (cerca de 17 mil bytes) e LONGTEXT (mais de 4 bilhões de caracteres). Para textos curtos, prefira VARCHAR.

Tipos para dados binários

  • BLOB: guarda dados binários, ou seja, informações expressas diretamente em bits; Util para armazenar imagens, videos ou outros arquivos de dados

Tipos para data e horas

  • DATE: armazena data no formato YYYY-MM-DD. Cada incremento representa um dia
  • TIME: Expressa tempo em formato HH:MM:SS
  • DATETIME exŕessa tempo no formato padrão YYYY-MM-DD HH:mm:SS
  • TIMESTAMP: ou o número de segundos passados desde a data de 01/01/1970 (data Unix). O valor é armazenado no formato 'YYYY-MM-DD hh:mm:ss' e vai de '1970-01-01 00:00:01' a '2038-01-19 03:14:07'. A diferença com relação ao DATETIME é que o valor de TIMESTAMP é influenciado pelo time zone (ou o fuso horário) configurado no servidor.

Chaves primárias e estrangeiras

O conceito de chave em um banco de dados consiste em ser um valor único em uma linha que distingue esta de todas as outras linhas da tabela. Esta seria, portanto, a chamada chave primária, que é usada para relacionar os dados de uma tabela com outra, sendo essa segunda chamada de chave estrangeira.

Definindo uma chave primaria:

  • Forma 1. Definindo a chave primária isoladamente

    CREATE TABLE nome_tabela (
   coluna_1 INT NOT NULL AUTO_INCREMENT
   coluna_2 VARCHAR(255)
   PRIMARY KEY(coluna_1)
   )

  • Forma 2. Definindo de forma declarativa

    CREATE TABLE nome_tabela (
   coluna_1 INT PRIMARY KEY NOT NULL AUTO_INCREMENT
   coluna_2 VARCHAR(255)
   )

  • Forma 3. Adicionando a chave primaria a posteriori

    ALTER TABLE nome_tabela ADD PRIMARY KEY (coluna_1);

Definição de uma chave estrangeira

  • As chaves estrangeiras são utilizadas para relacionar uma tabela com outra.

    O fundamento-base de um banco de dados relacional é sua capacidade de criar relacionamento entre tabelas, para que o administrador desse banco possa facilmente relacionar dados entre uma tabela e outra. Esses relacionamentos são gerenciados por chaves estrangeiras, que são, essencialmente, campos em tabelas que referenciam chaves primárias de outras tabelas.

    • IMPORTANTE: Não há chave estrangeira sem chave primária

    • Definindo uma chave estrangeira

      CREATE TABLE tabela1 (
   coluna_1 INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   coluna_2 VARCHAR(255),
   )
CREATE TABLE tabela2 (
   coluna_1 INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   coluna_2 VARCHAR(255),
   FOREIGN KEY tabela1_id REFERENCES tabela1 (coluna_1)
      • Dessa forma é criado um campo tabela1id na tabela2, que rerferencia o campo coluna1 da tabela1

  1. Relacionamentos em SQL

    1. Relacionando um item a outro item de forma um para um (1:1):

      • Em SQL o relacionamento 1:1 é simples: basta criar uma tabela com uma chave primaria e referenciar essa chave primaria na outra tabela como uma chave estrangeira que também é uma chave primária
      CREATE TABLE remedio (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   nome VARCHAR(255) NOT NULL,
   );
CREATE TABLE bula (
   dados VARCHAR(255) NOT NULL
   id_remedio INT NOT NULL PRIMARY KEY
   FOREIGN KEY (id_remedio) REFERENCES remedios(id) 
   )
      • Dessa forma criamos um relacionamento onde cada bula está diretamente associada a um remédio e só pode ser identeificada através do remédio, impedindo que mais de um remédio possua a mesma bula ou que uma bula se relacione com diversos remédios.

    2. Relacionando 1 item a n itens de forma 1 para n (1:N)

      • O relacionamento 1 para n também é bem fácil. Basta adicionar a chave primária de uma tabela como chave estrangeira de outra tabela

         CREATE TABLE clientes (
    id INT PRIMARY KEY NOT NULL AUTO_INCREMENT
    nome VARCHAR (255) NOT NULL
    CPF INT NOT NULL UNIQUE
    );
CREATE TABLE vendas (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT
   data_venda DATETIME NOT NULL
   valor DECIMAL(9,2) NOT NULL
   id_cliente INT NOT NULL
   FOREIGN KEY (id_cliente) REFERENCES clientes(id)
   )
        • Dessa forma uma venda é feita a apenas um cliente mas um cliente pode fazer várias compras criando um relcionamento de 1 para N

    3. Relacionando N itens a N itens de forma n para n (N:N)

      • No SQL não é possível criar um relacionamento N para N sem o auxilio de uma terceira tabela. Imagine que queremos incluir no banco anterior um campo que registra quais produtos foram vendidos em cada venda. Sera necessário criar uma nova tabela que lista os itens de uma determinada compra indiviualmente:

        CREATE TABLE vendas (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT
   data_venda DATETIME NOT NULL
   valor DECIMAL(9,2) NOT NULL

   )
CREATE TABLE produtos (
   id INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
   nome VARCHAR(255),
   valor DECIMAL(9,2),
   )
CREATE TABLE veda_produto (
   id_venda INT NOT NULL,
   id_produto INT NOT NULL,
   quantidade INT NOT NULL,
   PRIMARY KEY (id_venda, id_remedio),
   FOREIGN KEY (id_venda) REFERENCES vendas(id),
   FOREIGN KEY (id_produto) REFERENCES produtos(id)
   )
        • Por intermédio da tabela vendaproduto podemos fazer o relacionamento de n vendas com n produtos
