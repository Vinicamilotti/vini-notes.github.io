Segurança do banco de dados: usuários e permissões de acesso (GRANT e REVOKE, roles), encriptação de dados, SQL injection; integridade de dados

O conceito de segurança de computadores está diretamente relacionado ao de segurança da informação, incluindo não apenas a proteção de dados e informações, mas também a proteção de sistemas e bancos de dados.

Introdução

  • Caracteristicas básicas da segurança da informação:
    1. Confidencialidade
      • Manter privado o que precisa ser privado. Na prática consistem em controlar o acesso aos dados para que apenas usuários autorizados possam acessa-los e/ou modifica-los
    2. Integridade
      • Consistem em manter os dados limpos e imaculados, não só quando são carregados mas também quando são armazenados. Significa garantir que apenas usuários com permissão podem alterar os dados
    3. Disponibilidade
      • Garantia de que quando um usuário autorizado precisa acessar uma informação ele pode faze-lo. Seja garantindo que as redes e aplicativos funcionam, os protocolos de segurança não estão prejudicando a produtividade ou garantindo que os recursos estejam disponíveis uando surge um problema que precisa ser resolvido

Usuários e permissão de acesso

  • Ao instalar um banco de dados é criado um usuário root, que possui acesso a todo o banco. No entanto, em algum momento, pode ser necessário conceder acesso do banco de dados a outra pessoa ou aplicativo.

    Como o acesso ao banco de dados é necessário para o desenvolvimento do sistema, cabe ao administrador de banco de dados passar as credenciais de acesso para a equipe de desenvolvimento. Porém, fornecer as credenciais do usuário root pode comprometer todos os registros e as configurações do banco de dados[…]

  • Sendo assim, para conceder acesso a um banco de dados é necessário criar novos usuários

Criando usuários

  • A sintaxe básica para criar um novo usuário é a seguinte
CREATE USER 'usuario'@'endereco' IDENTIFIED BY 'senha'

    • endereco: Endereço de ip do banco de dados
    • senha: senha que o usuário deve utilizar para conexão com o banco

  • Exemplificando: criando um novo usuário no banco de dados local

Excluindo usuários:

  • Apesar de não ser recomendado é possivel fazer a exclusão de usuários. Veja a seguir a sintaxe básica:

    DROP USER 'usuario'@'endereco';

Permissões de usuário

  • Além de criar um usuário é necessário atribuir a ele privilégios.

    Note que são muitas as permissões que podem ser concedidas ou revogadas a um usuário, e a definição dessas permissões dependerá do tipo de usuário com o qual você estará trabalhando. Se você estiver criando uma conta de um administrador, por exemplo, o ideal é que ela tenha a maioria das permissões concedidas. Já para um usuário comum, algumas permissões de manipulação de dados e de manipulação de tabela já seriam suficientes.

  • Para conceder privilégios a um usuário utiliza-se o comando GRANT e para revogar privilégios o comando REVOKE

  • Considerando a segurança do banco de dados, é comum administradores criarem um novo usuário com todas as permissões e revogar os privilegios da conta root

    • Obs: já feito

  1. GRANT

    • A função GRANT pode conceder todos os privilégios que um dado usuário possui para outro usuário. Veja a sintaxe básica

      GRANT privilegios ON nome_banco.nome_tabela TO usuario@endereco;

    • Para garantir função de administrador a uma conta a sintaxe é um pouco diferente

      GRANT ALL PRIVILEGES ON *.* TO 'usuario'@'endereco'

    • Após conceder privilégios a um usuário é necessário atualizar as permissões do banco de dados. Para isso é utilizado o comando FLUSH:

      FLUSH PRIVILEGES;

    • Permissões mais comuns

      • Permissões de dados:
        • SELECT
        • INSERT
        • UPDATE
        • DELETE
      • Permissões de tabela
        • CREATE
        • DROP
        • ALTER

      • Exemplificando: Concedendo privilégios de dados para a conta teste criada anteriormente

        GRANT CREATE, SELECT, INSERT, UPDATE ON *.* TO 'teste'@'localhost'

      • Verificacndo:

        SELECT User, Create_priv, Select_priv, Insert_priv, Update_priv FROM mysql.user WHERE User = 'teste'

    • Observações: Perceba que o usuário 'teste' não possui permissão de deleção de dados, veja o que acontece se, logados como teste, tentamos deletar algum dado na tabela 'super' (tabela super foi criada na [UC03] mais precisamente em Consulta de dados)

      • Input:

        DELETE FROM Produto WHERE id=1

      • Output:

        ERROR 1142 (42000) at line 2: DELETE command denied to user 'teste'@'localhost' for table `super`.`Produto`

REVOKE

  • A instrução REVOKE faz revoga privilégios de uma conta. A sintaxe básica é parecida com a da instrução GRANT
REVOKE privilegios ON nome_banco.nome_tabela FROM 'usuario'@'endereco';
  • Exemplo: Retirando os privilégios da conta teste@localhost
REVOKE ALL PRIVILEGES ON *.* FROM 'teste'@'localhost';
FLUSH PRIVILEGES;
  • Verificando
SHOW GRANTS FOR 'teste'@'localhost'  
Grants for teste@localhost
GRANT USAGE ON *.* TO `teste`@`localhost` IDENTIFIED BY PASSWORD '*0971389F8E7F1AEB999104BDA7A0FA145087F348'

Roles

  • Role - função, papel - é uma coleção nomeada de privilégios. Cada role pode ter privilegios concedidos e revogado. Uma usuário pode receber uma role e, a partir dai, possuir os privilégios da role recebida.
    • Funciona como uma prédefinição de atributos para cada papel de usuário que consome o banco de dados.
    • Exemplo: 1 gerente e 2 funcionários do financeiro consomem o banco. Ao invés de dar privilegios um a um criamos 2 roles, uma para gerente, que posusi todos os privilégios, e uma para o financeiro, que possui apenas privilégios relacionados a area. Agora podemos atribuir essas roles a cada funcionário. Além disso se os funcionários mudarem podemos também atribuir as roles aos novos funcionários.

Criando uma role:

  • A sintaxe básica que criação de uma role é:

    CREATE ROLE 'nome_role_1', 'nome_role_2', 'nome_role_x'

  • Exemplo

    CREATE ROLE 'app_developer', 'app_read', 'app_write';

Atribuindo privilégios a cada role

  • Funciona da exata mesma forma que atribuir privilegios a usuário, no entanto n utilizamos o endereço. Sintaxe básica:

    GRANT privilegio ON database.tabela TO 'nome_role';

  • Exemplo: atribuindo privilégios diferentes as roles 'appdeveloper', 'appread', 'appwrite'

GRANT ALL ON app_db.* TO 'app_developer';
GRANT SELECT ON app_db.* TO 'app_read';
GRANT INSERT, UPDATE, DELETE ON app_db.* TO 'app_write';

Atribuindo Roles a usuários

  • Quando criamos uma role ela pode substituir os privilégio que queremos conceder a um usuário. Sintaxe básica:

    GRANT 'nome_role' TO 'usuario'@'localhost';

  • Exemplo: concedendo os privilégios devidos aos usuários criados

    GRANT 'app_developer' TO 'usuario_desenvolvedor'@'localhost';

GRANT 'app_write' TO 'usuario_escrita'@'localhost';

Encripitação de dados

A encriptação envolve a conversão de um texto simples e legível para humanos em um texto incompreensível, que é conhecido como texto cifrado. Essencialmente, isso significa pegar dados legíveis e alterá-los para que pareçam aleatórios.

Criptografia simétrica

  • Também conhecida como criptografia de chave privada.
  • Usa uma única chave para criptografar dados. A chave é um valor textual, como uma senha, que codifica e descodifica uma informação.
  • Na criptografia simétrica tanto a encriptação como desencriptação ocorrem através da mesma chave
  • Apesar de considerada menos segura a criptografia de chave privada utiliza pouco poder computacional
  • O tamanho da chave pode ser de 128 bits a 256 bits
  • O algoritimo mais utilizado é o AE5

Criptogrfia assimétrica

  • Também conhecida como par chave privada-publica.
  • Usa uma chave para encriptação e outra para desencriptação.
  • Apesar de mais lenta é mais segura, já que são necessárias duas chaves.
  • Possui a partir de 2048 bits
  • O algoritimo mais usado é o RSA

Funções de Hash

  • Funções de hash são unidimensionais e irreversiveis, ou seja, não é possível recuperar a mensagem original sem uma chave desencriptadora.
  • O Hashing é uma maneira de transformar um texto em outro teste.
  • A unica maneira de quebrar um hash é brute force
  • Util para manter informações sensiveis em bancos de dados.
  • Algoritimos de hash:
    • MD5
    • SHA-1
    • SHA-2

Criptografia no MYSQL

  • O MYSQL conta com algumas funções internas de segurança que incluem algoritimos de criptografia. Veja a tabela Mysql: Tabela de funções de criptografia
  • As funções mais utilizadas em Mysql são:
    • Hash:
      • MD5, SHA-1 e SHA2
    • Criptografia simérica:
      • AES

Utilizando MD5

  • O MD5 criará um texto criptografado de 128 bits (32 caracteres) independentemente do valor original.

  • Sintaxe básica:

    MD5('algum texto')

  • Exemplificando:

    1. Criando um usuário sem criptografia

      INSERT INTO cadastro_usuario (usuario, senha) VALUES ('Sem criptografia', 'Uma senha qualquer')

    2. Criando um usuário com criptografia

      INSERT INTO cadastro_usuario (usuario, senha) VALUES ('Com criptografia MD5', MD5('Uma senha qualquer'));

    3. Verificando:

      SELECT * FROM cadastro_usuario;
    • O MD5 faz criptografia de STRINGS portanto é imporante certificar de que os dados inseridos na função MD5() são do tipo correto
    • MD5 sempre retorna o mesmo hash para o mesmo texto, mas um completamente difirente para textos meramente diferentes;

      • Exemplificando:

        INSERT INTO cadastro_usuario (usuario, senha) VALUES ('MD5 mesma senha 1', MD5('Mesma senha'));
INSERT INTO cadastro_usuario (usuario, senha) VALUES ('MD5 mesma senha 2', MD5('Mesma senha'));
INSERT INTO cadastro_usuario (usuario, senha) VALUES ('MD5 meramente diferente', MD5('Mesma senho'));
    SELECT * FROM cadastro_usuario WHERE id IN (3, 4, 5)

Utilizando SHA-2

  • O SHA2 difere um pouco dos anteriores já que o algoritimo precisa que seja indicado o numero de bits que deve possuir o hash.

  • O numero de bits deve ser uma dessas opções

    • 224 bits : 28 caracteres
    • 256 bits : 32 caracteres
    • 384 bits : 48 caracteres
    • 512 bits : 64 caracteres

  • Sintaxe básica

    SHA2('texto', bits)

  • Exemplificando:

    INSERT INTO cadastro_usuario (usuario, senha) VALUES ('SHA2 com 224 bits', SHA2('Alguma senha', 224))
INSERT INTO cadastro_usuario (usuario, senha) VALUES ('SHA2 com 384 bits', SHA2('Alguma senha', 384));

    Diferentes das hashs MD5, SHA-1 e SHA-2, a criptografia AES gera valores binários com caracteres que não se limitam ao alfanumérico. Por essa razão, essa criptografia é considerada mais segura que o uso de hashs.

  1. Utilizando o AESDECRYPT()

    • Para desencriptar AES utilziamos a função AESDECRYPT()
    • A sintaxe básica segue a formula
    AES_DECRYPT(coluna_criptografada, 'chave')
    • Se executarmos a função diretamente no entanto temos um resultado estranho:
    SELECT AES_DECRYPT(senha_aes, 'Chave super secreta') FROM cadastro_usuario WHERE id = 10
    AESDECRYPT(senhaaes, 'Chave super secreta')
    BLOB
    • Isso ocorre pois o resultado continua no tipo VARBINARY e portanto deve ser convertido utilizando a função CAST()

      • A função CAST segue a seguinte sintaxe básica:

        SELECT CAST(Valor_de_um_tipo AS TIPO())

        • Exemplo simples:

          SELECT CAST(1 AS DECIMAL(3,2))
        CAST(1 AS DECIMAL(3,2))
        1.00
    • Dessa forma podemos utilizar o CAST para converter o tipo VARBYNARY em CHAR
    SELECT CAST(AES_DECRYPT(senha_aes, 'Chave super secreta') AS CHAR(200)) FROM cadastro_usuario WHERE id = 10
